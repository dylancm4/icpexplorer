## Context

The repository is empty (just `.git/` and `openspec/`). We need to scaffold a full-stack ICP application with a React frontend and Motoko backend. The project must follow the standard ICP project layout as established by the official `icp-cli-templates` hello-world template, while replacing the default tooling choices with our preferred stack (TanStack, Biome, Vitest, shadcn/ui).

The icp CLI (v0.2.1) uses `icp.yaml` at the project root and `canister.yaml` in each canister subdirectory. The `@icp-sdk/bindgen` Vite plugin generates TypeScript bindings from Candid `.did` files, and canister IDs are passed to the frontend via the `ic_env` cookie mechanism.

## Goals / Non-Goals

**Goals:**
- A working full-stack ICP app that can be built, tested, and deployed locally with `icp deploy`
- All tooling (linting, formatting, testing, routing, data fetching, styling) configured and working together
- SPA routing working correctly on the asset canister via `.ic-assets.json5`
- Internet Identity available in the local development environment
- A minimal landing page that proves the stack works end-to-end (frontend renders, can call backend canister)

**Non-Goals:**
- Implementing any application features, pages, or business logic
- Setting up CI/CD pipelines
- Configuring mainnet deployment (we'll handle this when we're ready to deploy)
- Choosing or configuring a chart library
- Implementing Internet Identity auth flow (just making II available locally for future use)

## Decisions

### Project directory structure

Follow the standard ICP layout with minor adjustments:

```
icpexplorer/
├── backend/
│   ├── src/
│   │   └── main.mo
│   ├── backend.did
│   ├── canister.yaml
│   └── mops.toml
├── frontend/
│   └── app/
│       ├── public/
│       │   └── .ic-assets.json5
│       ├── src/
│       │   ├── backend/
│       │   │   └── api/          (auto-generated by bindgen)
│       │   ├── components/
│       │   ├── routes/
│       │   ├── lib/
│       │   ├── main.tsx
│       │   └── routeTree.gen.ts  (auto-generated by TanStack Router)
│       ├── biome.json
│       ├── index.html
│       ├── package.json
│       ├── tsconfig.json
│       ├── tsconfig.app.json
│       ├── tsconfig.node.json
│       └── vite.config.ts
├── openspec/
├── .claude/
├── .gitignore
└── icp.yaml
```

**Rationale:** Matches the official hello-world template structure. Backend gets its own top-level `backend/` directory (not `motoko-backend/` since we're only using Motoko). Frontend lives in `frontend/app/` as the template prescribes. The `src/backend/api/` directory is auto-generated by `@icp-sdk/bindgen` and should be gitignored.

### Canister configuration

Use `canister.yaml` files in each canister directory rather than inlining configuration in `icp.yaml`. The root `icp.yaml` simply references the canister directories:

```yaml
canisters:
  - backend
  - frontend
```

Each canister directory contains its own `canister.yaml` with recipe configuration. This follows the pattern from the official templates and keeps configuration close to the code it describes.

### Vite + @icp-sdk/bindgen for TypeScript bindings

Use the `@icp-sdk/bindgen` Vite plugin to auto-generate TypeScript types and actor factories from the backend's `backend.did` Candid file. This eliminates manual type maintenance and ensures frontend types stay in sync with the backend interface.

**Alternative considered:** Manual type definitions — rejected because they'd drift out of sync with the Candid interface.

### TanStack Router with file-based route generation

Use TanStack Router's file-based routing with the Vite plugin (`@tanstack/router-plugin/vite`). Routes are defined as files in `src/routes/`, and the route tree is auto-generated. This provides type-safe navigation and parameters with minimal boilerplate.

**Alternative considered:** Code-based route configuration — rejected because file-based routing is the recommended approach and scales better as the app grows.

### shadcn/ui initialization

Initialize shadcn/ui with the "new-york" style (the more modern/polished variant) and configure it to use CSS variables for theming. This sets up the component infrastructure without adding any actual components yet — we'll add components as needed.

### Biome at the frontend/app level

Place `biome.json` in `frontend/app/` since that's where all the lintable/formattable code lives. Motoko has its own formatter. Keep the config minimal — Biome's defaults are sensible.

### Vitest configuration

Configure Vitest in `vite.config.ts` using the `test` property. Use jsdom as the test environment for React component testing. No test files yet — just the configuration ready to go.

## Risks / Trade-offs

**[Recipe version pinning]** The `canister.yaml` files pin specific recipe versions (e.g., `@dfinity/motoko@v4.1.0`, `@dfinity/asset-canister@v2.1.0`). These may not be the latest versions at the time of scaffolding. → Check the latest available versions when implementing.

**[@icp-sdk/bindgen compatibility]** The bindgen Vite plugin is relatively new. If we hit issues, we can fall back to manually running `icp build` and copying the generated types. → Start with the plugin, have a manual fallback plan.

**[shadcn/ui + TanStack Router]** shadcn/ui's CLI expects certain project structures. We need to verify it works correctly within the `frontend/app/` subdirectory rather than the project root. → Run `npx shadcn@latest init` from within `frontend/app/`.

**[Local II availability]** Setting `ii: true` in the network config starts a local Internet Identity canister. This adds startup time to the local dev environment. → Acceptable trade-off for having auth ready when we need it.
